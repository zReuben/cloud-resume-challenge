The Cloud Resume Challenge

How I built a full-stack, serverless, CI/CD-driven resume site on AWS

Intro

The Cloud Resume Challenge is a well-known learning project designed to test cloud fundamentals end-to-end. I used it as my starting point to practice AWS, infrastructure-as-code, CI/CD, and serverless. What began as a static website quickly expanded into a full production-like environment with multi-account separation, automated tests, and a promotion pipeline.

Front End: Static Website

  • HTML, CSS, and JavaScript for layout and styling.
  • Deployed to an S3 bucket and distributed globally via CloudFront.
  • HTTPS enforced with AWS Certificate Manager.
  • Custom domain reubenmulholland.com pointed using Route 53 DNS.
  • Visitor counter on the landing page calls a live API endpoint.

[TEXT PLACEHOLDER: snippet of index.html or style.css]

Back End: Serverless API + Database

  • API Gateway exposes RESTful endpoints for the counter.
  • AWS Lambda functions (Python + boto3) handle requests.
  • DynamoDB table stores persistent visitor count.
  • IAM roles scoped to least privilege; no over-permissive access.
  • CORS policies limit API calls to my domain.

Infrastructure as Code

  • Infrastructure defined in YAML using AWS SAM (Serverless Application Model).
  • CloudFormation stacks build S3, CloudFront, DynamoDB, API Gateway, and Lambda.
  • No manual console work — fully reproducible deployments.
  • MFA and AWS Vault used locally for secure role assumption.

[TEXT PLACEHOLDER: snippet of template.yaml showing DynamoDB + Lambda resource definitions]

CI/CD Pipeline

  • GitHub Actions workflow automates build → test → deploy.
  • Unit tests: Python (PyTest) for Lambda logic.
  • Integration tests: API Gateway + DynamoDB.
  • End-to-end tests: JavaScript checks visitor counter on live site.
  • Full test coverage required before deploy.
  • Artifact built once in ENG, promoted through TEST → PROD with approval and rollback.

Security

  • IAM roles and S3 bucket policies locked down.
  • No long-lived AWS keys: GitHub OIDC role assumption instead.
  • MFA enforced on all AWS accounts.
  • AWS Vault used locally — no secrets in source control.

[TEXT PLACEHOLDER: diagram of OIDC trust GitHub → AWS]

What I Learned

  • How front end and back end integrate in a serverless context.
  • The discipline of Infrastructure as Code with CloudFormation/SAM.
  • How DNS, SSL certificates, and CDNs actually fit together.
  • CI/CD as a confidence layer — tests that prevent bad deploys.
  • Practical experience with serverless (API GW, Lambda, DynamoDB).
  • Why multi-account separation (ENG/TEST/PROD) is a best practice.
