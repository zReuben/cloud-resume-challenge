Cloud Resume Challenge

How I built a full-stack, serverless, CI/CD-driven resume site on AWS

CloudResumeChallenge.dev Live Site GitHub ← Back to home

The Cloud Resume Challenge is a hands-on spec for proving cloud fundamentals end-to-end. I implemented it as a **full-stack build**: a static frontend, a RESTful API backend, and a database—defined with Infrastructure as Code and deployed via CI/CD pipelines.

Architecture at a glance

Front end

  • Static site (HTML/CSS/JS) hosted in S3, distributed via CloudFront.
  • HTTPS with ACM; custom domain reubenmulholland.com on Route 53.
  • Redirects from www → apex for SEO, consistent SSL, and simpler CORS rules.
  • Visitor counter displayed dynamically with JavaScript calling the API.

Back end (RESTful API)

  • API Gateway exposes RESTful endpoints (GET, POST, PUT, DELETE).
  • Lambda (Python + boto3) implements handlers and returns JSON responses with proper HTTP codes.
  • DynamoDB table persists the count (on-demand capacity, no over-provisioning).
  • CORS scoped only to my domain; IAM least-privilege enforced.
Everything as code. All infrastructure is declared in AWS SAM (CloudFormation) YAML templates. No console clicks—reproducible builds only.

Security & practices

  • MFA on all accounts; local role assumption via AWS Vault.
  • Secrets isolated per environment; nothing in source control.
  • Hardened IAM, scoped S3 policies, explicit CORS rules.
  • GitHub → AWS uses OIDC (no long-lived keys).
  • .gitignore prevents credential leaks; repo licensed under MIT.

What I learned

  • How the **full stack** fits together: frontend, RESTful API, database, IaC, CI/CD.
  • Best practices: single-purpose Lambdas, IaC without hardcoded resources, versioned JS for caching.
  • How DNS, ACM, and CloudFront integrate to deliver a production-grade site.
  • Running PyTest locally with virtualenv for parity with CI/CD pipelines.
  • Even small apps benefit from automation, least privilege, and reproducibility.
